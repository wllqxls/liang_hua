# liang_hua — 全自动量化交易机器人

## 项目目标

三阶段递进：**回测验证 → 模拟盘 → 实盘小仓位**。

MVP（第一阶段）：在本机浏览器打开网页，选币种、选策略、调参数，点击运行后在历史数据上回测，看到收益曲线和统计指标。

## 技术栈

| 层 | 选型 | 说明 |
|---|------|------|
| 语言 | Python 3.11+ | 全栈 Python，前端用模板 |

| 策略参数搜索 | 确定性渐进搜索 | 已实现粗筛、精搜、有限策略参数组和稳健性验证 |

## 当前入口模式

只允许以下信号模式作为 Web、API、优化器和验证脚本的主动入口：

- `KEY_LEVEL`
- `RSI_REVERSAL`
- `KEY_LEVEL_RSI`

保证金模式只允许：

- `ISOLATED`
- `CROSS`

`SRBreakout`、`MovingAverageCross`、`RSIReversion`、`KeyLevelScoring` 等旧类文件可以保留，但不能重新注册为主动入口策略。

## 研究候选：PULLBACK_CONFIRMATION

`PULLBACK_CONFIRMATION` 是研究专用候选，不是主动入口策略：

- 不得出现在 Web/API 的普通回测选项、参数优化器、策略诊断任务、testnet 或实盘路径。
- 旧的 `KEY_LEVEL`、`RSI_REVERSAL`、`KEY_LEVEL_RSI` 保持失败基线；禁止为了改善结果继续调它们的参数或改写报告。
- 候选必须用独立状态机处理：关键位事件 -> 最多 3 根后续 K 线确认 -> 一次入场 -> 等待价格离开事件位后才可发现新事件。
- 事件、确认、失效、重置和入场均只能使用该根 K 线已收盘的数据；事件本身不得在同一根 K 线上入场。
- 4h 环境只允许作为 `OFF` 与 `ALIGN` 两个预先声明的对照预设，默认 `OFF`；不得在研究完成后选择性地把它硬编码为必选条件。

候选只有在独立研究报告中同时满足现有验证标准及以下更严格门槛，才可提出升级为正式策略的设计变更：

- 成本后 Profit Factor >= `1.15`；
- 12 个独立窗口中至少 8 个窗口收益为正；
- 2025 和 2026 各自的成本后收益均为正；
- BTC/USDT 和 ETH/USDT 都完成验证，且不存在单一标的贡献全部正收益的情形；
- 一段未参与规则设计和参数选择的时间区间，仍通过完整验证。

候选的研究输出写入 `docs/research/`。新增该目录中的设计与报告文件时，必须注明数据范围、固定成本、参数预设、代码版本和是否属于保留样本。

## 因子事件研究

在提出下一条候选策略前，必须先完成只读的因子事件研究：

- 从原始 K 线提取关键位突破事件；每一行仅包含事件 K 线收盘时已知的特征，以及明确标记为事后标签的未来收益。
- 特征至少包括方向、突破 ATR 距离、K 线实体 ATR 距离、ATR 相对价格、RSI、成交量相对过去 20 根 K 线的比率、1h 环境和 4h 标签。
- 标签使用事件后固定持有期的方向收益：5m 入场周期为 `5m/15m/1h/4h`，15m 入场周期为 `15m/1h/4h`；标签不得回流到特征计算或事件筛选。
- 固定研究成本为一次完整往返的 taker 手续费和滑点，即 `2 * (0.0005 + 0.0002)`；4 小时以内标签不计资金费。
- 报告必须按方向、4h 标签、成交量分组和波动率分组展示样本数、平均毛收益、平均成本后收益和胜率。样本数不足 200 的桶不得形成策略结论。
- 事件 CSV 是可再生的派生产物，写入 `results/research/` 并忽略 Git；研究报告写入 `docs/research/` 并纳入 Git。
- 发现某个桶成本后收益为正，只能标为“待验证因子”；只有在跨年份、跨标的和保留样本重复成立后，才能提出新的候选策略设计。
- 独立事件假设可使用与关键位不同的事件状态机，但必须在设计文档中冻结事件 A、事件 B、等待窗口、单次触发和重置规则；不得复用策略入场逻辑或据结果补充筛选条件。
- 波动率极值收敛后的突破研究固定使用 5m ATR(14)、其 20 期均线及已收盘的 1h Bollinger(20, 2) 上下轨/中轨；事件 B 只在事件 A 之后的预设窗口内首次触发，产生方向性未来收益标签。事件 A 本身没有交易方向，报告应展示其到事件 B 的转化率而不能虚构盈亏比。
- 极端动量后的均值回归研究固定使用同一根已收盘 5m K 线可见的 RSI(14) 和 Bollinger(20, 2)。每段连续带外极端行情只记录首次 A；A 后仅检查下一根已收盘 K 线是否回到中轨另一侧。上轨极端的收益标签方向为 `SELL`，下轨极端为 `BUY`。若 A→B 转化率低于 10%，或任一年度/标的 A 样本不足 200（尤其只有几十个），报告必须明确否决该假设，不得生成策略。
- 成交量吸收反转研究必须复用 `src/research/event_factors.py` 与 `scripts/research_event_factors.py`，不得创建另一套研究台。放宽版事件 A 固定为成交量相对前 20 根均量 >= `1.5`、真实波幅/ATR(14) <= `1.0`、过去 3 根收盘位移绝对值 >= `1.0 ATR`；研究方向与位移相反。事件 B 固定为随后 3 根内达到 `0.5 ATR` 的反向移动。BTC/ETH 2024/2025 必须同时跑 5m 与 15m 平行对照，不得看完一个周期后修改事件逻辑。
- 短期趋势惯性研究必须复用现有因子研究台。5m 与 15m 分别独立检测：连续 3 根收盘价变化同号，3 根累计收益绝对值 >= `0.003`，每段同向序列只记录首次达标事件；研究方向与序列方向相同。B 固定为事件后 `5m/15m/1h` 的同向毛收益是否大于 0，成本后收益统一扣除完整往返 `0.0014`。15m 事件的 5m 标签必须来自同步 5m 收盘数据，不能用 15m 数据插值。
- 1h 极端动量快速反转研究固定使用 BTC/ETH 2024/2025 的 1h K 线。事件 A 为以下并集：连续两根同色实体且每根实体幅度 >= `0.005`、两根从首根开盘到次根收盘累计幅度 >= `0.015`；或当前实体幅度 >= `0.005` 且收盘越过 Bollinger(20,2) 对应外轨。每段同向极端只记录首次 A，研究方向与极端方向相反。B 为随后第 1 或第 2 根收盘相对 A 收盘达到 `0.5 * ATR(14)` 的反向移动。报告只统计 1h/2h，固定完整往返成本为 `0.0014`。
- BTC→ETH 短期领先效应研究固定使用同步的 BTC/ETH 2024/2025 5m K 线，2026 作为未使用保留期。BTC 最近 3 根收盘位移按 BTC ATR(14) 标准化，绝对值必须 >= `1.0`；ETH 同期必须同方向，且按 ETH ATR(14) 标准化后的绝对位移不超过 BTC 的 `0.5`。每段同方向候选只记录首次事件，研究 ETH 沿 BTC 方向的 `5m/15m/30m/1h` 精确收盘收益，单腿完整往返成本固定为 `0.0014`。主检验窗口预先固定为 `15m`；2024 和 2025 都必须样本 >= 200、成本后均值 > 0、PF >= 1.15，且日历日区块 bootstrap 的成本后 95% 置信区间下界 > 0，才允许提出策略候选。

## 编码约定

### Python
- 类型注解：所有函数参数和返回值必须有类型注解
- 命名：函数/变量用 `snake_case`，类用 `PascalCase`
- 字符串：单引号用于代码内字符串，双引号用于文档字符串和用户可见文本
- 文件头：不需要编码声明和 shebang（除非是入口脚本）
- 日志：用 `logging` 模块，不用 `print`
- 错误处理：捕获具体异常类型，不用裸 `except`
- 配置：所有可配置项走 `pydantic-settings` 或环境变量

### 前端
- HTML/CSS/JS 全部手写，不引入前端框架
- Chart.js 通过 CDN 引入
- JS 用原生 DOM API，不引入 jQuery
- CSS 用 CSS 变量管理颜色主题
- 所有 API 调用加 loading/error 状态处理
- `/` 使用同一 DOM 内的左右双页面：左侧为回测，右侧为策略诊断；通过边缘方向按钮平滑切换，切换时不得刷新页面或清空已填写状态
- 策略诊断必须是独立页面区域，不嵌套在回测面板内；桌面和窄屏都必须能从两侧来回切换
- 回测页的“本地数据”折叠面板内部固定为二级双页面：`基础行情` 与 `订单流数据`。二级切换只能移动面板内部内容，不得改变外层“主界面/策略诊断”页面、URL 或用户已填写状态；年份选择由两个二级页面共享。
- `基础行情` 保留原有单标的年度拉取和状态表。`订单流数据` 以 BTCUSDT 与 ETHUSDT 作为同一个年度研究包，必须显示后台任务阶段、已完成文件数、总文件数、错误和逐标的完整性状态，不能用一个长 HTTP 请求阻塞页面。

### 数据
- K 线数据 CSV 列：`timestamp, open, high, low, close, volume`
- 时间戳用 ISO 8601 字符串（`2024-01-01T00:00:00`）
- 新数据文件路径：`data/{year}/{SYMBOL}_{TIMEFRAME}.csv`，如 `data/2025/BTC_USDT_1h.csv`
- Web 服务和本地 Python 默认数据根目录固定为项目根目录下的 `data/`，不能随启动终端目录变化
- 网页数据管理只主动拉取 `5m`、`15m`、`1h`、`4h`
- 点击拉取指定年份数据时必须一次性拉取上述 4 个周期
- 同一年同一周期重复拉取时必须合并新旧 CSV，按时间戳去重、排序并覆盖保存
- 根目录旧文件如 `data/BTC_USDT_1h.csv` 可以保留兼容，但不是新功能主动入口
- SYMBOL 中 `/` 替换为 `_`
- 信号模式回测必须从所选 `data_year` 目录读取入场周期、`1h`、`4h` 已收盘上下文数据
- 策略验证脚本可合并 `data/{year}/` 年份目录生成 `tmp/validation_data/` 临时验证数据，用于覆盖 `365 天 + warmup`
- `data/` 下行情文件由程序写入，git 忽略，不纳入提交
- Binance USDⓈ-M 订单流数据固定写入 `data/order_flow/binance_um/{raw,normalized}/{dataset}/{symbol}/{year}/`。原始 ZIP 必须先校验官方 `.CHECKSUM`，标准化文件按 UTC 5m 桶输出；下载器不得读取或修改 `.env`，不得把原始大文件加入 Git。
- 第一阶段订单流数据只允许增强 `5m klines`、`aggTrades` 校验样本、`metrics`、`fundingRate` 和只读 `bookDepth` 样本审计。正式 5m 主动买卖量优先使用官方增强 K 线中的 taker buy 字段，避免无必要地下载两年约 28 GB 压缩逐笔数据；`aggTrades.isBuyerMaker=true` 必须解释为主动卖出，`false` 为主动买入。官方未提供的历史爆仓数据不得用 K 线推测或伪造，只能标记缺失或从未来开始实时采集。

- Binance `metrics.create_time` 允许偏离标准时点数秒，标准化时必须先向下归入对应 UTC 5m 桶，再跨全年合并、去重和审计；不得按单日文件边界误判为年度缺口。官方确实缺失的 5m 桶必须保留为空值并设置 `metrics_available=false`，禁止长时间前向填充或伪造；因子研究必须排除特征窗口内包含缺口的样本。

## 策略验证规则

`scripts/validate_strategies.py` 必须覆盖 3 个信号模式，固定使用更保守的 `ISOLATED` 作为策略验证基准。

保证金模式不是策略有效性的重复验证维度。`ISOLATED` 与 `CROSS` 的信号、止盈和止损完全相同，差异只在强平抵押资金；两者的强平差异由模拟器单元测试和独立风险压力测试验证，不能复制整套年度策略验证。

每个组合必须运行：

- 12 个不重叠的 30 天窗口
- 1 个 365 天窗口
- 固定使用 taker 手续费率 `0.0005`、单次成交滑点率 `0.0002`、每 8 小时资金费率 `0.0001`

只有同时满足以下阈值才可标记为 `通过`：

- 平均窗口收益为正
- 最差窗口收益大于 `-40%`
- 全年收益为正
- 最大回撤小于 `30%`
- Profit Factor 大于等于 `1.05`
- 年化交易次数大于等于 `50`

未通过验证的模式保持不可用于未来自动化 testnet 执行。

每次完整验证还必须生成失败诊断，至少包含：

- 成本前收益、手续费、资金费净现金流、成本后净收益
- 成本前与成本后 Profit Factor、胜率和平均每笔净收益
- 按出场原因、交易方向、1 小时环境和 4 小时过滤标签拆分的交易次数与净收益
- 基于客观统计生成失败原因，不允许用诊断结果放宽上述通过阈值

### Web 策略诊断

- 普通回测、页面启动和页面切换不得自动运行完整策略诊断
- 用户必须在策略诊断页主动启动 365 天诊断；启动后由后台任务自动完成 3 个信号模式
- 前端必须显示任务阶段、已完成组合数、总组合数、耗时和错误状态，并轮询后台任务，不能让单次 HTTP 请求阻塞到验证结束
- 同一进程同时只允许一个策略诊断任务运行，重复启动必须返回明确提示
- 最新诊断结果以结构化 JSON 保存到 `results/strategy-diagnostics.json`，页面打开时自动读取最近结果
- 结构化结果必须来自生成 Markdown 报告的同一次年度回测，不允许为前端重复运行回测或解析 Markdown 表格
- 策略诊断页至少展示 3 个信号模式的验证状态、核心收益/成本指标、跨模式结论和每组主要失败原因

## 开发流程

1. 新功能先在 `src/` 写核心逻辑
2. 加对应的测试
3. 在 `src/web/routes.py` 加 API 端点
4. 在 `templates/` 和 `static/` 加前端页面
5. 验证：`pytest` 全绿 + 浏览器手动测试
6. 每次更改完成并验证通过后，主动创建内容明确的 Git commit
7. `git push` 必须先取得用户本次明确授权；禁止 force push

## 禁止事项

- 不要在代码里硬编码 API key
- 不要在回测中使用未来数据（look-ahead bias）
- 不要注释掉测试让它们通过
- 不要直接修改 `data/` 目录下的 CSV 文件（只允许程序写入）
- 不要在没有回测验证的情况下直接实盘
